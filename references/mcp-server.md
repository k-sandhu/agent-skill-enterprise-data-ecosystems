# Read-Only MCP Server

Every built ecosystem can ship a generated, self-contained, **read-only MCP server** that exposes the SQLite database, the schema/lineage knowledge in the spec, the imperfection log, controls and DQ rules, and the documentation artifacts to any MCP client (Claude Code, Claude Desktop, or any stdio MCP host). An agent that receives a generated package can connect to it as a first-class MCP server and explore the data, lineage, and embedded knowledge without knowing anything about this repo.

## Purpose

- The server is **generated, never hand-written** — `scripts/generate_mcp_server.py` emits it from the spec plus the build outputs. This matches the engine philosophy: workers and agents do not hand-write mechanical artifacts.
- It is **strictly read-only**, with five independent safety layers (below) and no write tools of any kind.
- All exposed data is **synthetic and fictional**, generated by the agent-skill-enterprise-data-ecosystems engine. It must not be treated as real organizational data or real PII. The disclaimer is baked into the server's `initialize` instructions, every prompt, and the README.
- It is **Python 3.9+ standard-library only** — no `mcp`/FastMCP SDK, no third-party packages, no network access. The server implements the MCP stdio transport (newline-delimited JSON-RPC 2.0) directly.

## When to generate

- Generate it by **default** for the "Complete local SQLite ecosystem" and "Full package" output modes — any time the user wants a portable, populated database.
- **Skip** it for brief-only or design-only outputs where no database is built (there is nothing to serve).
- **Regenerate after every rebuild.** Row counts, the imperfection log, and the resource list change with each build, and the manifest captures them. Rerun the generator (with `--force`) whenever you rebuild the database.

## Generator CLI

```text
python scripts/generate_mcp_server.py <spec.json> --build <dir> [--db <path>] [--name <server-name>] [--force] [--quiet]
```

- `--build` — the build output directory (the one passed to `build_sqlite_ecosystem.py --out`). The package is emitted to `<build>/mcp/`.
- `--db` — database path. Default: `<build>/<physical_name(org)>.db`; otherwise the sole `*.db` in `<build>`; otherwise the generator exits 1 asking you to disambiguate. The database must exist and open read-only.
- `--name` — server name. Default: `physical_name(org_name).replace("_", "-") + "-data-ecosystem"` (e.g. `harborline-provisions-data-ecosystem`).
- `--force` — required to overwrite an existing `<build>/mcp/` package (mirrors the build script's `--force`).
- `--quiet` — suppress progress output.

Exit codes: 0 success, 1 generation failure, 2 usage error.

### What it emits

Exactly three files into `<build>/mcp/`:

| File | Contents |
| --- | --- |
| `server.py` | A byte-for-byte copy of `scripts/mcp_server_template.py`. |
| `mcp_manifest.json` | Ecosystem-specific knowledge: tables, lineage edges, controls, DQ rules, resources, and limits (sorted keys, `\n` newlines, no timestamps or absolute paths). |
| `README.md` | A connection guide tailored to this build. |

It also writes a clean, BOM-free, note-stripped copy of the spec to `<build>/ecosystem_spec.json` so the `ecosystem://spec` resource travels with the package by relative path. The manifest references the database and every documentation artifact by **package-relative path**, so the `mcp/` directory plus its parent build directory relocate as a unit.

Determinism is a contract: the three emitted files are byte-identical across runs, machines, and `PYTHONHASHSEED` values. The self-test proves this with a double-generation byte comparison.

## Tool surface

Every tool returns a single text content item whose text is a JSON object (`json.dumps(..., sort_keys=True)`). Table arguments accept the logical key `schema.name` (e.g. `erp.customer`) or the physical name (e.g. `erp_customer`); an unknown table returns `isError: true` listing the closest matches.

| Tool | Args | Returns |
| --- | --- | --- |
| `get_ecosystem_overview` | — | Organization, time horizon, seed/multiplier/engine version (read live), table count per layer, total rows, and counts of imperfections, controls, and DQ rules, plus the fictional-data disclaimer. |
| `list_tables` | `schema?`, `layer?` | Each table's logical key, physical name, layer, purpose, grain, source, and live row count. Optionally filtered. |
| `describe_table` | `table` | Keys, traits, source system, and every column with type, nullability, sensitivity, and FK target. |
| `sample_rows` | `table`, `limit?` (1–100, default 10) | First rows in deterministic `rowid` order: `{columns, rows, row_count}`. |
| `query` | `sql`, `limit?` (1–1000, default 200) | Read-only single SELECT/WITH. `{columns, rows, row_count, truncated}`; `truncated` reflects the row limit or the 1 MB byte cap. |
| `get_lineage` | `table`, `direction?` (`upstream`/`downstream`/`both`) | Transitive lineage from the spec's derivations and dataflows; each entry is `{table, via}`. |
| `list_imperfections` | `table?`, `type?`, `limit?` (1–100, default 20) | `{total, by_type, by_table, examples, note}` from `meta_imperfection_log`; `limit` caps examples only. |
| `get_table_profile` | `table` | Live row count and per-column null %, distinct count, and min/max. Distinct counts are skipped for tables over 2,000,000 rows. |
| `list_controls` | — | The spec's controls plus the validation result views they map to. |
| `list_dq_rules` | — | The spec's data-quality rules. |
| `get_build_info` | — | `meta_build_info` contents, engine version, database file size, and the SQLite library version. |

## Resources and prompts

Resources use the `ecosystem://` URI scheme; only files that exist on disk at call time are listed. `resources/read` returns `{"contents": [{uri, mimeType, text}]}`.

| URI | File | mimeType |
| --- | --- | --- |
| `ecosystem://spec` | `ecosystem_spec.json` | `application/json` |
| `ecosystem://docs/build-summary` | `build_summary.json` | `application/json` |
| `ecosystem://docs/validation-report` | `validation_report.md` | `text/markdown` |
| `ecosystem://docs/profile` | `profile.md` | `text/markdown` |
| `ecosystem://docs/schema-catalog` | `schema_catalog.md` | `text/markdown` |
| `ecosystem://docs/server-readme` | `mcp/README.md` | `text/markdown` |
| `ecosystem://ddl/<basename>` | each `sqlite/*.sql` (e.g. `ecosystem://ddl/01_schema`) | `application/sql` |

Two prompts are provided:

- `tour-this-ecosystem` (no arguments) — walks the client through `get_ecosystem_overview`, `list_tables` per layer, and `sample_rows` on a mart and a fact table, then a synthetic-data-flagged summary.
- `investigate-imperfections` (optional `table`) — `list_imperfections`, pick two types, use `query` to find affected rows, check `list_dq_rules`/`list_controls` for the catching rule, and trace propagation with `get_lineage`.

## Safety model

The server is read-only by construction, with five layers applied in order on every query:

1. **Read-only connection** — `sqlite3.connect("file:...?mode=ro", uri=True)`, opened once and reused.
2. **Deny-by-default authorizer** — only `SQLITE_SELECT`, `SQLITE_READ`, `SQLITE_FUNCTION`, and `SQLITE_RECURSIVE` are allowed; `ATTACH`/`DETACH`/`PRAGMA` and every write/DDL action are denied. Installed once at startup and never cleared while serving.
3. **Statement vetting** (for `query`) — comments are stripped, the first keyword must be `SELECT` or `WITH`, and multi-statement input is rejected (`SQLite`'s own lexer finds the first statement boundary, so a `;` inside a string literal does not count).
4. **Query timeout** — a progress handler aborts any tool call that exceeds 5 seconds of wall-clock time; the tool returns `isError: true` mentioning the limit.
5. **Row/byte caps** — per-tool row clamps plus a 1 MB response byte cap; `query` reports `truncated`.

A tool or SQL failure is **not** a JSON-RPC protocol error — it returns a normal `tools/call` result with `isError: true` and an explanatory message. The server loop never crashes on bad input or bad SQL. **There are no write tools. Period.**

## Connecting clients

Substitute the real absolute path to `server.py` for the placeholder in each block.

**Claude Code:**

```sh
claude mcp add <server-name> -- python /absolute/path/to/this/directory/server.py
```

Or a project-scoped `.mcp.json`:

```json
{"mcpServers": {"<server-name>": {"command": "python", "args": ["/absolute/path/to/this/directory/server.py"]}}}
```

**Claude Desktop:** add the same `mcpServers` block to `claude_desktop_config.json`. On Windows, write paths with forward slashes or doubled backslashes.

**Generic stdio host:** run `python server.py` with the `mcp/` directory as the working directory.

## Limits

| Limit | Value |
| --- | --- |
| Default query rows | 200 |
| Max query rows | 1000 |
| Max sample rows | 100 |
| Max response bytes | 1,000,000 |
| Query timeout (seconds) | 5 |

## Troubleshooting

- **`python` not found (Windows):** use `py` instead of `python` in the connect command, or the full interpreter path.
- **Path escaping in JSON:** in `.mcp.json` / `claude_desktop_config.json`, Windows paths need forward slashes or doubled backslashes.
- **Server exits immediately:** the manifest or database could not be found. Run the server directly with `--verbose` to see the error on stderr; check that `mcp_manifest.json` sits next to `server.py` and that the database is where the manifest's relative path points.
- **Database moved:** keep the `mcp/` directory and the build directory in the same relative positions (the db path is relative), or pass `--db` to point at the database elsewhere.
- **Client shows no tools:** something other than the protocol is writing to stdout. The server writes protocol bytes to stdout exclusively and logs only to stderr — make sure no wrapper injects output.
- **Stale row counts or missing resources:** regenerate the package after rebuilding (`--force`); the manifest is a snapshot of the build it was generated from.
